# Wany KB — Skills Framework

A composable capability layer for the Wany agents. Inspired by
[obra/superpowers](https://github.com/obra/superpowers) — the idea that an
agent is not a monolith but a **runtime that loads skills on demand**, the
same way an OS loads drivers.

> "Skills are the composition unit of agents."
> — Superpowers, 2026

In Wany KB, a skill is a single directory containing a `SKILL.md` file with
a declarative frontmatter contract and a prose body of instructions. Any
Wany agent (Norma, Finn, Corpus, and future additions) can install,
version, and compose skills by pointing at this directory. The viewer
exposes the registry over HTTP so downstream routers — including the local
Agent Router on `localhost:9000` — can discover what the repo can do.

---

## Directory layout

```
skills/
├── README.md              ← this file
├── _registry.json         ← generated by bootstrap/install-skills.sh
├── norma/                 ← legal / policy agent
│   └── SKILL.md
├── finn/                  ← finance / metrics agent
│   └── SKILL.md
└── corpus/                ← knowledge-base / research agent
    └── SKILL.md
```

Each skill lives in its own directory. A skill MAY add an `assets/`
subfolder with reference prompts, templates, golden examples or any other
file the body of `SKILL.md` wants to reference.

## SKILL.md schema

Every `SKILL.md` is a Markdown file with a YAML frontmatter block. The
frontmatter is the machine-readable contract; the body is the
human/LLM-readable instructions. The frontmatter conforms to this JSON
Schema:

```json
{
  "$schema": "http://json-schema.org/draft-07/schema#",
  "title": "WanySkill",
  "type": "object",
  "required": ["name", "agent", "description", "inputs", "outputs"],
  "properties": {
    "name":        { "type": "string", "pattern": "^[a-z0-9-]+$" },
    "agent":       { "type": "string", "enum": ["norma", "finn", "corpus"] },
    "version":     { "type": "string", "default": "0.1.0" },
    "description": { "type": "string", "maxLength": 280 },
    "inputs": {
      "type": "array",
      "items": {
        "type": "object",
        "required": ["name", "type"],
        "properties": {
          "name":        { "type": "string" },
          "type":        { "type": "string" },
          "required":    { "type": "boolean", "default": true },
          "description": { "type": "string" }
        }
      }
    },
    "outputs": {
      "type": "array",
      "items": {
        "type": "object",
        "required": ["name", "type"],
        "properties": {
          "name":        { "type": "string" },
          "type":        { "type": "string" },
          "description": { "type": "string" }
        }
      }
    },
    "examples": {
      "type": "array",
      "items": {
        "type": "object",
        "properties": {
          "title": { "type": "string" },
          "input": { "type": "object" },
          "output": { "type": "object" }
        }
      }
    },
    "tags": { "type": "array", "items": { "type": "string" } }
  }
}
```

### Required fields

| field         | why                                                              |
|---------------|------------------------------------------------------------------|
| `name`        | lowercase slug, unique per agent, used as registry key           |
| `agent`       | which Wany agent owns this skill                                 |
| `description` | one-sentence summary (<= 280 chars) shown in the router          |
| `inputs`      | ordered list of parameters the skill consumes                    |
| `outputs`     | shape of the data the skill produces                             |

### Optional fields

| field      | why                                                                |
|------------|--------------------------------------------------------------------|
| `version`  | semver, defaults to `0.1.0`                                        |
| `examples` | concrete input/output pairs — the LLM learns the contract here    |
| `tags`     | routing hints (`legal`, `finance`, `pipeline:stow`, ...)           |

## Writing a good skill

1. **One skill, one verb.** If the `name` needs an "and", split it.
2. **Be specific in `description`.** "Review a contract" is bad;
   "Flag unusual clauses in SaaS MSAs against Wany policy" is good.
3. **Type your inputs.** Use JSON-Schema-ish primitives (`string`, `number`,
   `markdown`, `url`, `file[]`). The router uses these to validate calls.
4. **Ship at least one `example`.** The example doubles as a smoke test
   and as few-shot context for any LLM loading the skill.
5. **Prose body = instructions.** Write the body of `SKILL.md` as if you
   were onboarding a new hire: preconditions, steps, edge cases, done
   criteria. Keep it under ~300 lines.

See `skills/norma/SKILL.md` for a reference implementation.

## Bootstrap & registry

```bash
bash bootstrap/install-skills.sh
```

What it does:

1. Walks `skills/*/SKILL.md`.
2. Parses the YAML frontmatter of each file.
3. Writes `skills/_registry.json` — a flat array of `{ name, agent, path,
   description, inputs, outputs, version, tags }` records, sorted by name.
4. Prints `Registered N skills: [names]`.

The script is **idempotent**: rerunning it produces a byte-identical
`_registry.json` (fields are sorted; the top-level `generated_at` field is
omitted from the hash). Fail modes:

- missing frontmatter      → skill skipped with a warning
- required field missing   → skill skipped with a warning
- unknown `agent`          → skill skipped with a warning

None of these are fatal; the script exits `0` so it can be chained from
`npm postinstall` or a git hook.

## `/api/skills` endpoint

The viewer (`viewer/server.js`) exposes:

```
GET /api/skills
→ 200 { ok: true, count: N, skills: [...] }
→ 500 { ok: false, error: "<msg>" }
```

The handler reads `skills/_registry.json` from disk on each call (the file
is small; no cache required). If the registry is missing, the endpoint
returns `{ ok: true, count: 0, skills: [] }` and logs a hint to run
`bootstrap/install-skills.sh`.

This endpoint is the MCP-style surface the Agent Router talks to. No UI is
wired yet — that is deliberate; the endpoint is the contract.

## Adding a new skill

1. `mkdir skills/<agent>/<skill-slug>` — or place `SKILL.md` directly under
   `skills/<agent>/` if the agent only has one skill.
2. Copy the frontmatter from an existing `SKILL.md` and edit.
3. Write the prose body.
4. Run `bash bootstrap/install-skills.sh` to re-generate the registry.
5. Commit `skills/<agent>/.../SKILL.md` **and** `skills/_registry.json`.
